Learn Tab Restructure — Detailed Implementation Plan

Context

The current Videos/Feed tab is a flat TikTok-style video scroller with inline MCQs. Games live on a completely separate tab launched by the teacher. This restructure replaces the Feed tab with a Learn tab where each module becomes a scrollable digital workbook: video → key takeaways →
quiz → game round. Tony handles backend + frontend scaffolding; Eman handles styling/polish.

---

1.  Expand TikTokModule Model — Key Takeaways + Linked Game Round

Requirement (PRD)

▎ "Key takeaways block (new). New interactive page document layout with 3 to statements boxes summarising the core facts from the video. Written in bite-sized, youth-accessible language. This has tts and dyslexic overlay features as well. Exists as a comprehension safety net so
▎ students who zone out during the video still absorb the key points before the quiz."

▎ "Game round block. A specific game round linked to this module's topic. Completing the quiz makes the game round playable. The game round is thematically tied to the module content."

Requirement (Transcript)

▎ "We need to add, you know, summary text. Bite-sized chunks that really, you know, add onto the video, and also like help summarise it." (3:02)

▎ "So now what we've done is actually we're gonna have to make and upgrade the games to be more modular by design. So not just the videos, but the games are modular. And we've kind of done this as well, quite well, with some of the games, so Vape Invaders has different rounds. We can
▎ actually toggle specific rounds inside the learn tab." (5:30)

Why This Change

The TikTokModule struct currently only holds Title, Description, VideoKeys, and Questions. There is no field for summary takeaways and no way to associate a specific game round with a module. Without these fields, the Learn tab cannot render the workbook flow or know which game to
embed after the quiz.

2.  Add Round Grouping to Chainlink

Requirement (PRD)

▎ "Chain Link: Add round grouping in CMS. Questions get assigned to rounds. Engine accepts roundId and plays only that round's questions. Category tags already exist so mapping is straightforward to the tags in a way and a round could contain specific tags e.g whats in a vape,
▎ environment tab in part 1"

Requirement (Transcript)

▎ "The box, you know how we have different challenge rounds, and we have alert and more basic rounds, they can, they're gonna have to be a bit more modular by design where we have a bit of a challenge round, and we can toggle on the harder questions onto that as a round." (7:04)

Why This Change

Chainlink currently has a flat Questions array with an optional Category field on each question. There is no concept of "rounds" — you either play all questions or nothing. For the Learn tab to embed "Chain Link Round 1: Chemicals", we need a way to group questions into named rounds
and request only that round's questions via the API.

Code Change

File: backend/internal/cms/models.go (lines 29-34)

Current:
type ChainlinkGame struct {
Questions []ChainlinkQuestion `json:"questions"`
TwoTruthsLieSets []TwoTruthsLieSet `json:"twoTruthsLieSets,omitempty"`
MysteryCardSettings MysteryCardSettings `json:"mysteryCardSettings"`
UpdatedAt time.Time `json:"updatedAt"`
}

Proposed:
type ChainlinkGame struct {
Questions []ChainlinkQuestion `json:"questions"`
Rounds []ChainlinkRound `json:"rounds,omitempty"`
TwoTruthsLieSets []TwoTruthsLieSet `json:"twoTruthsLieSets,omitempty"`
MysteryCardSettings MysteryCardSettings `json:"mysteryCardSettings"`
UpdatedAt time.Time `json:"updatedAt"`
}

// ChainlinkRound groups a subset of questions into a named round
// that can be played independently via the Learn tab
type ChainlinkRound struct {
ID string `json:"id"`
Title string `json:"title"`
QuestionUUIDs []string `json:"questionUuids"` // references ChainlinkQuestion.UUID
IsActive bool `json:"isActive"`
}

The flat Questions array remains the source of truth. Rounds reference questions by UUID. When Rounds is empty, all questions form one implicit round (backward compatible with current behavior). The existing Category field on questions can be used as a shortcut when initially assigning
questions to rounds in the CMS.

API Filtering

File: backend/internal/api/routes.go (or new learn_handlers.go)

When the content endpoint is called with a ?roundId=xxx query parameter, filter the Questions array to only those whose UUIDs appear in the matching ChainlinkRound.QuestionUUIDs. This is API-level filtering — no game engine changes needed.

3.  Add Round Grouping to TheBox

Requirement (PRD)

▎ "The Box: Same pattern as Chain Link. Add round grouping. Statements in quick sort and challenge rounds can be grouped to rounds. Sorting mechanic is unchanged. But in learn tab CMS we can configure different rounds of games"

Why This Change

TheBox already has implicit round assignments via RoundAssignment ("sort_1", "sort_2") and ChallengeRound ("challenge_1", "challenge_2") fields on cards. However, there is no formal round list — you can't reference "TheBox Round 1" by ID. For the Learn tab to link a module to a
specific Box round, we need a typed round list that maps round IDs to the existing assignment keys.

Code Change

type TheBoxRound struct {

}

Add to TheBoxConfig:

When Rounds is empty, derive them from the existing card assignments (backward compat). When a roundId is provided via API, filter ClearCutCards by RoundAssignment and AmbiguousCards by ChallengeRound matching the selected round's RoundKey.

4.  New Learn Block Completion Action Type

Requirement (PRD)

▎ "MCQ gating and progression: Content below the video (takeaways, quiz, game round) is blurred and locked until the video is marked complete."

▎ "Modules unlock sequentially. Module 1 is available when the teacher enables navigation. Module 2 unlocks when Module 1 is complete (video watched, quiz attempted)."

Requirement (Transcript)

▎ "Once the student watches the videos, they see the text, they see the puzzle, but then the rest of the content is blurred as well. So once they complete the puzzle and they get it right, they can then move on to the next section which becomes unblurred." (3:54)

Why This Change

The current ActionModuleInteraction action type bundles video watch + quiz response into a single action submitted at module completion. The Learn tab needs granular per-block tracking — we need to know independently when a video was watched, when takeaways were read, when a quiz was
passed, and when a game round was played. This drives the unlock/gating logic and the teacher's module progress view.

5.  Learn Flow Content Endpoint

Requirement (PRD)

▎ "The Videos tab is renamed to Learn. Inside it, each module is a single scrollable page with content blocks stacked vertically: Video block, Key takeaways block (new), Quiz block, Game round block."

Why This Change

The frontend currently calls GET /api/content/:module/modules/:keyStage/:sessionNumber to get TikTokModuleCollection (modules with videos + questions only). The Learn tab needs the full workbook structure including resolved game round metadata (title, game type). Rather than making the
frontend call 4+ separate endpoints and stitch them together, a single composite endpoint returns everything needed to render the Learn flow.

Code Change

New file: backend/internal/api/learn_handlers.go

// LearnFlowResponse is the composite response for the Learn tab
type LearnFlowResponse struct {
Modules []LearnModule `json:"modules"`
UpdatedAt time.Time `json:"updatedAt"`
}

type LearnModule struct {
ID int `json:"id"` // 1-based
Title string `json:"title"`
Description string `json:"description"`
Videos []string `json:"videos"` // resolved video URLs
KeyTakeaways []cms.KeyTakeaway `json:"keyTakeaways"`
Questions []cms.TikTokModuleQuestion `json:"questions"`
LinkedGame \*ResolvedLinkedGame `json:"linkedGame,omitempty"`
}

type ResolvedLinkedGame struct {
GameType string `json:"gameType"`
RoundID string `json:"roundId"`
RoundTitle string `json:"roundTitle"` // resolved from the game's config
}

Handler implementation:

1.  Call cmsService.GetTikTokModulesForModule(ctx, module, keyStage, sessionNumber)
2.  For each module with a LinkedGame, fetch the relevant game config and resolve the round title
3.  Return the composite response

Route registration in backend/internal/api/routes.go:
contentGroup.GET("/:module/learn-flow/:keyStage/:sessionNumber", handlers.GetLearnFlow)

7.  Points Redistribution

Requirement (PRD)

▎ "Points redistribution. Each session in a module contains around 900 to 1100 points and stays the same. Game points (currently 300 per game) get split across the number of game rounds in the game. Quiz points (300 total) get split across module. So if its 6 video learn modules is
▎ 300/6 per round. If its 5 video modules its 300/5. So each way points are equal across sessions."

Requirement (Transcript)

▎ "The point system, it'll be a lot more then to think about how we organise the points system and make it a bit more, again modular within these different sessions." (8:13)

Why This Change

Currently game points are awarded as a flat bonus per complete game (e.g., 300 for playing all of Chain Link). With games split into rounds across the Learn flow, points need to be distributed per-round so each module's game block awards a proportional share. Quiz points similarly need
splitting so each module quiz contributes equally.

Code Change

File: backend/internal/points/ (config + calculator)

// Per-module quiz points = 300 / number of modules in session
// Per-round game points = game's total points / number of rounds in session
// First-time completion only — retries don't award points
// Skill-based scoring within games (VapeInvader accuracy, etc.) remains unchanged

The calculator reads the learn_block_complete action's attemptNumber field — only attemptNumber == 1 awards points, matching the PRD:

▎ "If they get the answer right first time they earn full points... Second attempts do not award points."

12. Frontend: Game Components Accept roundId

Requirement (PRD)

▎ "For this to work, every game needs to support being called with a specific round."

▎ "Vape Invaders: Already modular with themed rounds. No engine changes. Just wire round IDs to Learn tab modules."
▎ "Spot The: Already modular with Easy, Medium, Hard rounds. No engine changes. Just wire round IDs to Learn tab modules."
▎ "Chain Link: Engine accepts roundId and plays only that round's questions."
▎ "The Box: Sorting mechanic is unchanged. But in learn tab CMS we can configure different rounds of games."

Why This Change

When a student clicks "Play Round" from the Learn tab or Games tab, the game component needs to filter to only that round's content. VapeInvader and SpotThe already have round arrays in their configs — they just need to read roundId from URL params and filter. Chainlink and TheBox need
to filter based on the new round grouping.

Code Change

Files: Game components read roundId from useSearchParams():

- VapeInvaderGame.tsx — filter config.rounds to matching roundId (or play all if no param)
- SpotTheGame.tsx — filter config.rounds to matching roundId
- ChainlinkGame.tsx — when roundId present, API returns pre-filtered questions (see section 2)
- TheBoxGame.tsx — when roundId present, API returns pre-filtered cards (see section 3)

No game engine logic changes — just data filtering at the input level.
